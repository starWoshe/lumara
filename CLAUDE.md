## 📐 Архітектурні принципи

### Головне правило
**Кожне рішення має бути масштабованим.**
Перед написанням коду запитай себе: "Чи витримає це 100 000 користувачів?"

### Структура монорепо
```
lumara/
├── apps/
│   ├── web/          # Next.js головний застосунок
│   └── admin/        # Адмін панель
├── packages/
│   ├── agents/       # AI агенти (LUNA, ARCAS, NUMI, UMBRA)
│   ├── content/      # Контент-генератор для соцмереж
│   ├── database/     # Prisma схема + міграції
│   ├── ui/           # Спільні компоненти
│   └── shared/       # Типи TypeScript, утиліти
├── CLAUDE.md
├── README.md
└── package.json
```

### Технічний стек
| Шар | Технологія | Призначення |
|-----|-----------|-------------|
| Frontend | Next.js 14 (App Router) | Головний застосунок |
| Styling | Tailwind CSS + shadcn/ui | UI компоненти |
| Auth | NextAuth.js | Авторизація |
| Database | Supabase (PostgreSQL) | База даних |
| ORM | Prisma | Робота з БД |
| AI | Claude API + GPT-4 | Агенти |
| Payments | Stripe | Монетизація |
| Email | Resend | Транзакційні листи |
| Deploy | Vercel + GitHub Actions | CI/CD |
| DNS/CDN | Cloudflare | Захист і швидкість |

## 🔄 Правила розробки

### Git workflow
1. **Кожен логічно завершений блок** → коміт → пуш → деплой
2. Назви комітів — українською, чіткі і конкретні:
   - ✅ `додано базову схему бази даних`
   - ✅ `реалізовано авторизацію через NextAuth`
   - ❌ `fix`, `update`, `changes`
3. Гілки:
   - `main` — продакшн (автодеплой на Vercel)
   - `dev` — розробка
   - `feature/назва-фічі` — нові функції

### GitHub Actions CI/CD
- Кожен пуш в `main` → автоматичний деплой на Vercel
- Кожен пуш в `dev` → preview деплой на Vercel
- Перед мержем в `main` → перевірка типів TypeScript

### Коміт після кожного блоку
```bash
git add .
git commit -m "опис що зроблено"
git push origin main
```

## 🗄️ База даних — Supabase PostgreSQL

### Принципи схеми
- Всі таблиці мають `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`
- Всі таблиці мають `created_at` і `updated_at`
- М'яке видалення: `deleted_at TIMESTAMP` (ніколи не видаляй записи)
- Мультимовність: окрема таблиця `translations`
- Мультитенантність: поле `tenant_id` для майбутнього white-label

### Основні таблиці
```
users           — користувачі
profiles        — профілі користувачів
agents          — AI персонажі академії
conversations   — сесії з агентами
messages        — повідомлення в сесіях
subscriptions   — підписки (Stripe)
courses         — курси академії
enrollments     — записи на курси
content_queue   — черга контенту для соцмереж
```

## 📁 Правила написання коду

### TypeScript
- Строгий режим: `"strict": true` в `tsconfig.json`
- Типи для всіх функцій і змінних
- Інтерфейси для всіх об'єктів БД

### Компоненти
- Server Components за замовчуванням
- Client Components тільки коли потрібна інтерактивність (`'use client'`)
- Компоненти в `/components/`, сторінки в `/app/`

### API Routes
- Всі API маршрути в `/app/api/`
- Завжди валідуй вхідні дані через `zod`
- Завжди повертай типізовані відповіді

### Змінні середовища
- Ніколи не хардкодь ключі в коді
- Всі секрети в `.env.local` (локально) і Vercel Environment Variables (продакшн)
- `.env.example` завжди актуальний

## 📋 Правило INSTRUCTIONS.md — обов'язкове для всіх модулів

**Перед реалізацією будь-якого нового модуля або агента — створи `INSTRUCTIONS.md` у його папці.**

### Структура INSTRUCTIONS.md
1. **Призначення** — що робить цей модуль
2. **Технічний стек** — які технології використовує
3. **Правила поведінки** — як генерує контент / як відповідає
4. **Інтеграції** — з якими модулями взаємодіє
5. **Змінні середовища** — які ключі потрібні
6. **Приклади** — вхідні та вихідні дані

### Порядок дій при створенні нового модуля
1. Створи папку модуля
2. Створи `INSTRUCTIONS.md` з повним контекстом
3. Закомітити: `додано INSTRUCTIONS.md для [назва модуля]`
4. Запушити на GitHub
5. Тільки після цього — починай реалізацію
